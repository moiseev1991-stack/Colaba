# Нагрузка и ресурсы: оценка пропускной способности

Краткий справочник по модели нагрузки поиска и аудита доменов, оценке времени выполнения и рекомендациям по CPU/RAM и количеству воркеров.

## Модель нагрузки

- **Один поиск** состоит из:
  - 1 задача `execute_search_task` (поисковая выдача);
  - N задач `process_domain_task` по числу уникальных доменов (типично 50–100).
- **Среднее время на домен** после оптимизаций (max_pages=10, timeout=20 с, лимит 5 мин на домен): ориентир **30–90 с** для обычных сайтов, до **5 мин** для самых медленных.
- **Параллелизм:** один Celery worker с `--concurrency=C` обрабатывает C доменов одновременно. Несколько воркеров или больший C увеличивают пропускную способность.

## Формула времени поиска

Ориентировочное среднее время завершения одного поиска:

```
время_поиска ≈ (число_доменов / суммарный_параллелизм) × среднее_время_на_домен
```

- **Суммарный параллелизм** = сумма `concurrency` по всем воркерам, обрабатывающим очередь `celery`.
- Пример: 80 доменов, 1 worker с concurrency=8, ~1.5 мин на домен → 80/8 × 1.5 мин ≈ **15 мин**.

## Рекомендуемые ресурсы по сценариям

| Сценарий              | Одновременных поисков | CPU      | RAM  | Примечание |
|-----------------------|------------------------|----------|------|------------|
| Dev / малая нагрузка  | 1–2                    | 2–4 ядра | 4 GB | Текущий dev setup |
| Prod малая            | 2–5                    | 4 ядра   | 8 GB | 1 worker celery concurrency=8, backend 2 workers |
| Prod средняя          | 5–15                   | 8 ядер   | 16 GB| 2 инстанса celery-worker по concurrency=8 или 1 с concurrency=16 |
| Prod высокая          | 15–50                  | 16+ ядер | 32 GB| Несколько celery-worker, Redis/Postgres на отдельном хосте или с лимитами |

## Postgres

- Соединения: `(uvicorn_workers × 2) + (celery_workers × concurrency)` и выше при пиках.
- Нужен достаточный `max_connections` в Postgres и настройка pool size в приложении под эту сумму.

## Redis

- Брокер сообщений и result backend для Celery.
- При тысячах задач в очереди учитывать память; результаты можно хранить с TTL.
- Ориентир: **256 MB–1 GB** в зависимости от длины очереди и TTL.

## Масштабирование при высокой нагрузке

При **десятках и сотнях одновременных запросов** (поисков) необходимо:

- **Горизонтальное масштабирование:** несколько инстансов `celery-worker` (или несколько машин).
- По необходимости: **ограничение длины очереди** или **приоритеты очередей**, чтобы тяжёлые сценарии не блокировали остальных пользователей.

Дальнейшие улучшения (параллельный обход страниц внутри домена, кэш robots.txt, приоритеты) — после замеров на выбранной конфигурации (см. [observation-search-audit-timing.md](../observation-search-audit-timing.md)).

## Проверка после изменений (re-observe)

После внедрения параметров (max_pages=10, timeout=20, лимит 5 мин на домен, concurrency=8 в prod):

1. Пересобрать и запустить воркеры с новым кодом:  
   `docker compose up -d --build backend celery-worker` (для prod — пересобрать образы и перезапустить соответствующие сервисы).
2. Запустить стек (если ещё не запущен): `docker compose up -d` (или `docker compose -f docker-compose.prod.yml up -d`).
3. Выполнить один поиск (например, «МФЦ Кемерово») через UI или API.
4. Просмотреть логи воркера:  
   `docker compose logs celery-worker` (или для prod: логи сервиса `celery-worker`).
5. Убедиться:
   - задачи по доменам завершаются не более чем за ~5 минут (нет 12+ минут на один домен);
   - при долгих доменах в логах есть предупреждение `crawl domain=... exceeded 5 min, saving partial result`;
   - среднее время на домен уменьшилось по сравнению с замером до оптимизаций (см. [observation-search-audit-timing.md](../observation-search-audit-timing.md)).
