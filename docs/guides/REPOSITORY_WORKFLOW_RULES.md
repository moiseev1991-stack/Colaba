# Правила работы с репозиторием Colaba

## Версия: 2.0 | Дата: 18 марта 2026

---

## 1. Git: коммиты

Формат: `<type>(<scope>): <description>`

| Тип | Описание |
|-----|----------|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `docs` | Документация |
| `refactor` | Рефакторинг |
| `test` | Тесты |
| `chore` | Конфигурация, зависимости |
| `style` | Форматирование |
| `perf` | Производительность |
| `ci` | CI/CD |
| `build` | Сборка |

Примеры:
- `feat(backend): Add OAuth authorization`
- `fix(frontend): Fix search form validation`
- `docs(deployment): Update Coolify guide`

## 2. Git: push

**Push -- ТОЛЬКО по явной команде пользователя.** Никогда автоматически.

После коммита: сообщить пользователю и спросить, делать ли push.

## 3. Git: ветки

Формат: `<type>/<name>`
- `feature/user-authentication`
- `fix/search-validation`
- `docs/api-documentation`
- `refactor/project-structure`

## 4. Git: что не коммитить

- `.env` (только `.env.example`)
- `__pycache__/`, `*.pyc`, `node_modules/`, `.next/`
- Лог-файлы, временные файлы
- IDE конфигурация (`.vscode/`, `.idea/`)
- Одноразовые отчёты и результаты проверок

## 5. Документация: обновлять, а не плодить

- Сделали фичу -- обновите `docs/STATUS.md`
- Запланировали задачу -- обновите `docs/ROADMAP.md`
- Запись о деплое -- добавьте в `docs/deployment/WORKLOG.md`
- Новое руководство для пользователя -- создайте в `docs/guides/`

Не создавайте одноразовые файлы типа `*_SUMMARY.md`, `*_REPORT.md`, `*_RESULT.md`. Обновляйте существующие.

## 6. Чеклист перед коммитом

- [ ] Файлы в правильных папках
- [ ] Нет мусорных/временных файлов
- [ ] Сообщение коммита соответствует формату
- [ ] Нет секретов в коде
- [ ] Документация обновлена (если нужно)
