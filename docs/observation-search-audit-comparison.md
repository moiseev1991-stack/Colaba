# Сравнение: до и после оптимизации поиска и аудита

## Источники данных

- **До:** замер по логам одного прогона «МФЦ Кемерово» — документ [observation-search-audit-timing.md](observation-search-audit-timing.md).
- **После:** фактический замер по поиску «МФЦ Кемерово» (провайдер Yandex XML, concurrency=6). Первый прогон (поиск 6) был с **num_results=25** в скрипте — получилось 33 домена, т.е. **в 3 раза меньше**, чем в замере «до» (105). Для честного сравнения запуск с **num_results=100** (поиск 7, **103 домена** — тот же порядок, что и 105). Провайдер по умолчанию — **Yandex XML** (DuckDuckGo удалён).

## 1.1 Скорость фазы поиска (получение выдачи)

Фаза поиска — это `execute_search_task`: запрос к провайдеру (Yandex XML, Яндекс HTML, Google HTML), получение списка URL и сохранение результатов в БД. Дальше идёт уже фаза аудита (crawl по доменам).

| Аспект | До | После |
|--------|-----|-------|
| **Yandex XML** | Загрузка страниц выдачи по одной (10 результатов за запрос) | **Параллельная загрузка батчами по 3 страницы** (`PAGE_BATCH_SIZE = 3` в `tasks.py`) — меньше суммарное время на 30–100 результатов |
| **Таймаут фазы поиска** | Не регламентирован явно | **120 с** на всю операцию получения выдачи (`asyncio.wait_for(..., timeout=120.0)` для не‑XML провайдеров; при превышении — понятная ошибка пользователю) |
| **Провайдер по умолчанию** | DuckDuckGo | **Yandex XML** — обычно быстрее и стабильнее: один API-запрос на 10 результатов, без парсинга HTML и без риска капчи/блокировок |
| **Фактический замер** | — | Поиск 6: **execute_search_task ~23 с** (Yandex XML, 25 результатов → 33 уникальных домена) |

Итого: скорость «поиска» (получение выдачи) улучшена за счёт параллельной загрузки страниц для Yandex XML, таймаута 120 с и перехода на Yandex XML по умолчанию.

## 3. Общее время одного поиска

- **До:** при ~20 доменах в замере и concurrency=4 общее время сильно определяли redmap.ru и yoplace.ru (**нижняя оценка ~12 мин** только на них); итог по наблюдению — **10+ минут** на один поиск.
- **После (ожидание):**  
  - формула: `(число_доменов / concurrency) × среднее_время_на_домен`;  
  - пример: 80 доменов, concurrency=8, ~1.5 мин на домен → **80/8 × 1.5 ≈ 15 мин**;  
  - при 6 слотах (dev) и том же среднем времени — **~20 мин** на 80 доменов.  
  Нет «хвоста» по 12 мин от одного-двух доменов — все ограничены 5 мин.

1. **backend/app/queue/tasks.py**

## 5. Повторный замер (при необходимости)

Фактический замер «после» уже выполнен (поиск 6, см. п. 2.2). Для повторного замера:

1. Пересобрать и запустить воркеры с текущим кодом:  
   `docker compose up -d --build backend celery-worker`
2. Запустить один поиск (например «МФЦ Кемерово») через UI или API с провайдером **Yandex XML** (в настройках провайдеров должны быть указаны folder_id и API-ключ).
3. Дождаться завершения поиска (статус `completed`).
4. Собрать логи:  
   `docker compose logs celery-worker > celery_logs_after.txt`
5. Из логов вытащить:
   - **Время на домен:** строки вида  
     `Task process_domain_task[...] succeeded in X.XXs`  
     (или `crawl domain=... in X.XXs, pages=N`).
   - **Ограничение по 5 мин:** строки  
     `crawl domain=... exceeded 5 min, saving partial result`.

По этим данным можно заполнить таблицу «после» в этом документе и при необходимости скорректировать max_pages/timeout/concurrency (см. [CAPACITY_AND_LOAD.md](guides/CAPACITY_AND_LOAD.md) и [observation-search-audit-timing.md](observation-search-audit-timing.md)).

## 7. Результаты и что дальше

### Итог оптимизации

- **Было:** один поиск на ~105 доменов упирался в 2 «тяжёлых» домена по 12+ минут; общее время 10+ минут и непредсказуемо.
- **Стало:** ни один домен не тянет дольше 5 минут (лимит + частичный результат); при ~100 доменах общее время ~10–15 минут при concurrency=6, без хвоста в 12 минут.
- **Вывод:** текущей оптимизации **достаточно**, если один поиск на 100 доменов за 10–15 минут и 2–5 одновременных пользователей вас устраивают.

### Дальше оптимизировать?

| Ситуация | Действие |
|----------|----------|
| **Один–несколько пользователей, 100 доменов за 10–15 мин ок** | Ничего не делать. Следить по логам за таймаутами и «exceeded 5 min». |
| **Нужно быстрее один поиск** | Поднять concurrency (например 8–10 в dev, 12–16 в prod) или добавить второй инстанс celery-worker. См. [CAPACITY_AND_LOAD.md](guides/CAPACITY_AND_LOAD.md). |
| **Много одновременных поисков (10+)** | Масштабировать: несколько celery-worker, при необходимости лимиты очереди и приоритеты. |
| **Хочется ещё ускорить crawl по домену** | Делать только после замеров: параллельный обход страниц внутри домена, кэш robots.txt, вынос max_pages/timeout в конфиг. |

**Рекомендация:** сейчас **не оптимизировать дальше**. Оставить текущие параметры, при росте нагрузки или требований — увеличить concurrency/воркеры и при необходимости вернуться к точечным доработкам (параллельный crawl, кэш).
