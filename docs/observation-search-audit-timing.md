### 2. Параллелизм

- **concurrency: 4** (prefork) — одновременно обрабатываются 4 домена.
- Обе очереди (`search_queue`, `celery`) обслуживает один worker.

### 3. Узкие места по логам

1. **Отдельные «тяжёлые» домены**  
   redmap.ru и yoplace.ru дают по ~12+ минут на домен из‑за множественных таймаутов (Timeout for ... attempt 1/3, 2/3, 3/3) и обхода до 20 страниц. Один такой домен сильно увеличивает общее время аудита.

2. **Последовательный crawl внутри домена**  
   В логах видна череда HTTP Request: GET ... — страницы одного домена обрабатываются подряд; при 20 страницах и таймаутах по 30 с время на домен легко достигает минут.

3. **Таймауты и ретраи**  
   Частые сообщения вида: `Timeout for https://yoplace.ru/... (attempt 1/3)` — до 30 с на попытку × 3 попытки на страницу. На «медленных» сайтах это даёт десятки секунд на одну страницу.

4. **Общее время аудита**  
   При 4 воркерах общее время определяется самыми долгими задачами. Два домена по ~12 мин дают нижнюю оценку ~12 мин только на них; плюс поиск и остальные домены — итог порядка 10+ минут на один поиск совпадает с наблюдением.

## Рекомендуемые следующие шаги (точечно)

- **Ограничить время/глубину по домену**  
  Уменьшить `max_pages` (например до 10) и/или общий таймаут на crawl одного домена, чтобы «тяжёлые» сайты не тянули общее время.

- **Уменьшить таймаут на одну страницу**  
  Снизить timeout с 30 с до 15–20 с для одного HTTP-запроса, чтобы быстрее отсекать очень медленные страницы.

- **Увеличить параллелизм**  
  Поднять `--concurrency` для celery-worker (например до 8), чтобы быстрее обрабатывать очередь доменов.

- **Защита от «долгих» доменов**  
  Ввести мягкий лимит времени на одну задачу `process_domain_task` (например task_soft_time_limit) или ограничить число страниц/запросов на домен, чтобы один сайт не занимал 12+ минут.

Дальнейшую оптимизацию (параллельный crawl страниц внутри домена, кэш robots.txt и т.д.) имеет смысл делать после внедрения этих шагов и повторного замера.
