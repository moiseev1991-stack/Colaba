## 1. Что такое Colaba

Colaba — B2B lead-generation платформа под российский рынок 2026. Архитектурно это конвейер:

> **поиск (карты + сайты) → краулинг → SEO-аудит / отзывы → AI-анализ болей → извлечение контактов → email-рассылка → парсинг ответов (IMAP) → AI-ассистенты → CRM-light**

плюс модуль госзакупок (`tenders`, не достроен) и multi-tenancy с RBAC.

**Главная фишка 2026:** не плоский список контактов как у Snov.io/Apollo, а **компании с диагнозом** — для каждой компании из 2GIS вытащены боли её клиентов из отзывов через LLM (sentiment + embeddings + кластеризация), привязаны к тегам болей. Цель — давать пользователю не «вот 1000 стоматологий», а «вот 23 стоматологии Москвы, у которых клиенты в отзывах за последние 3 месяца жаловались на грязь — вот цитаты — вот контакты — вот черновик письма».

## 2. Контекст автора и ограничений

- **Дима** — solo-фаундер. Без команды и без маркетингового бюджета.
- Рынок РФ 2026 — падающий: и у клиентов, и у автора нет денег на ads.
- Канал привлечения должен быть бесплатным: контент-SEO, Telegram-канал «прогонов», партнёрка 30%.
- Ниша на квартал **де-факто пошла по картам + отзывы**: пайплайн maps + reviews_ai отстроен, остальное на паузе.
- Технически автор грамотный: понимает SPF/DKIM, прогрев домена, Postfix, Celery — упрощать не надо.

## 3. Технологический стек

### Backend
- **Python 3.11** + **FastAPI 0.104** (async)
- **SQLAlchemy 2.0** (async) + **Alembic** (актуальная ревизия — **032**)
- **PostgreSQL 16** (порт 5433 локально → 5432 в контейнере) + **pgvector** extension (для embeddings отзывов)
- **Redis 5** + **Celery 5.3** (6+ воркеров, очереди: search, crawl, audit, contacts, outreach, replies, maps, reviews_ai)
- **SQLAdmin 0.19** — встроенная админка (**auth добавлена — v1.49.2**)
- **slowapi** (rate limiting — пока не включён в prod)
- **JWT** (`python-jose`) + **bcrypt**, httpOnly cookies
- **fastapi-sso** — OAuth (Google, Yandex, VK, Telegram Login Widget)

### Frontend
- **Next.js 14.2** (App Router) + TypeScript 5.3
- **React 18.2**, **Tailwind CSS 3.4**
- **@tanstack/react-query 5.17**, **zustand 4.4**, **axios 1.6**, **lucide-react**
- Production-режим (`next build` + `next start`), порт **4000**

### Внешние интеграции и сервисы
- **Поиск по картам:** 2GIS Catalog API (платный ключ Димы) + 2GIS public widget API (бесплатный fallback для отзывов)
- **Поиск по сайтам:** DuckDuckGo (free) + Yandex XML/HTML + Google HTML — с фолбеком
- **Yandex Cloud Search API** (folder_id + API-ключ) — через `yandex-cloud-ml-sdk`
- **LLM через ProxyAPI** (РФ-шлюз к OpenAI/Anthropic): `OPENAI_BASE_URL=https://api.proxyapi.ru/openai/v1`. Используется gpt-4o-mini для sentiment и naming, text-embedding-3-small для embeddings отзывов.
- **Email:** Hyvor Relay (SMTP-инфра РФ-friendly) + IMAP catch-all `reply-{user_id}@domain` для парсинга ответов
- **Captcha bypass:** AI Vision + 2captcha + anticaptcha + Yandex SmartCaptcha

### DevOps
- Docker Compose (dev / prod / GHCR)
- GitHub Actions: CI (тесты), deploy, semantic-release (**Deploy workflow периодически падает — есть ручной путь через Coolify Terminal**)
- Conventional Commits + commitlint + husky
- **Прод:** VPS 88.210.53.183 → nginx → Traefik → Coolify-приложение `Colaba` (`/opt/colaba/docker-compose.prod.yml`), домен `spinlid.ru`
- **Известная грабля:** фронт нельзя собирать на VPS (3.8GB RAM, OOM убивает весь стек). Только локальный `docker build` фронта → `docker save | scp | docker load`.

## 5. Что уже реализовано в коде

### Backend-модули (`backend/app/modules/`)
- `auth/` — JWT, регистрация, OAuth (Google / Yandex / VK / Telegram реализованы, **credentials в проде есть не для всех**)
- `searches/` — поиск с фолбеками, SearchProvider абстракция, краулер сайтов + extract контактов (email/телефоны)
- `providers/` — конфиги провайдеров поиска (DuckDuckGo / Yandex / Google)
- `filters/` — пользовательские фильтры по результатам
- **`maps/`** — парсер карт (2GIS), компании + отзывы. SSE-стрим прогресса парсинга. Самоисцеление битого кэша, авто-перепарс отзывов при `from_cache` для компаний с `reviews_count=0`. Города: верифицированный `region_id` есть для Москвы, остальные через fallback `region_id=70000001` (вся Россия).
- **`reviews_ai/`** — LLM-пайплайн: sentiment (gpt-4o-mini, батчинг по 20), embeddings (text-embedding-3-small), кластеризация болей, LLM-naming тегов. **Подключён к ProxyAPI**, работает в проде.
- `email/` — кампании, шаблоны, домены, IMAP-парсинг ответов, **DB-backed mail config** через `email_config` singleton + UI `/app/email/settings`
- `outreach/` — генерация писем, отправка через `EmailService` (Hyvor Relay)
- `ai_assistants/` — CRUD ассистентов, chat / vision endpoints, реестр моделей. Сидируются ассистенты `reviews_ai_sentiment`, `reviews_ai_naming` (миграция 017).
- `captcha/` — обход капчи (4 движка)
- `organizations/` — multi-tenancy + RBAC (OWNER / ADMIN / MEMBER)
- `tenders/` — модуль госзакупок (**заложен, не достроен, на паузе**)
- `payments/` — заложен, биллинг не интегрирован
- `dashboard/`, `monitor/` — статистика и наблюдение
- `deployments/` — история деплоев + Deployments API

### Frontend-разделы (`frontend/app/`)
`auth/`, `dashboard/`, `leads/` (включая `proposals/` — шаблоны КП, и новый UI карт с фильтрами по болям/рейтингу/отзывам, drawer «Все отзывы», pill-метрики), `seo/`, `tenders/`, `organizations/`, `profile/`, `settings/` (AI, провайдеры, капча, деплои, email), `monitor/`, `payment/`, `policy/`, `runs/`.

В корне `app/` есть `error.tsx` и `global-error.tsx`. API-прокси через `frontend/app/api/v1/[...path]/route.ts` → `INTERNAL_BACKEND_ORIGIN` (по умолчанию `http://127.0.0.1:8001`).

## 6. Что НЕ работает / блокеры до первых продаж

Из `docs/Plan_2026-05-03.md` + текущие блокеры:

1. **Self-service биллинг** — ЮKassa / CloudPayments + рекуррентка + лимиты тарифов **enforced в коде**.
2. **Onboarding за 60 секунд** — пресеты появились, но полноценного тура нет.
3. **Доменная инфра для cold email** — SPF / DKIM / DMARC + warm-up + ротация доменов + bounce / complaint handling. Hyvor Relay есть, но UI для 5 доменов «под отправителя» нет.
4. **Demo-аккаунт / песочница** — без захода внутрь B2B-клиент в РФ 2026 не зарегается.
5. **2–3 публичных кейса** — «было / стало», скриншоты, цифры.
6. **Rate limiting** — slowapi есть, в prod не включён.
7. ~~Обрыв пайплайна maps→outreach~~ ✅ **ЗАМКНУТ** — контакты, цитаты, драфт письма, листы лидов — всё работает.

### Что вырезать (для solo это важнее, чем добавить)
- OAuth VK и Telegram Login Widget — B2B так не логинятся. Оставить Email + Yandex.
- AI-ассистентов как раздел — это внутренний инструмент, спрятать в шестерёнки.
- WhatsApp в roadmap — для РФ B2B не нужен (Meta-блок).
- Госзакупки и SEO-аудит — две разные продуктовые сущности. Карты + отзывы выиграли по приоритету.

## 7. Стратегический выбор (де-факто закрыт)

**Ниша квартала:** **2GIS-парсинг с диагнозом болей**. По убыванию приоритета ICP:

| ICP | Почему | Готовый UVP |
|---|---|---|
| **Маркетологи / агентства локального бизнеса** | Их клиенты — рестораны/фитнес/стоматологии, которым нужны лиды | «Вот 50 потенциальных клиентов в твоём городе, к каждому есть конкретная боль в отзывах — продай им SMM/SEO под эту боль» |
| **B2C-сервисы продающие в B2B** (POS, CRM для общепита, бухучёт для салонов) | Их ICP — компании из 2GIS | «Найди в своём городе 200 ресторанов с проблемой долгого обслуживания — продай им систему очередей» |
| **Конкурентная разведка** | Локальный бизнес хочет видеть боли конкурентов | «Что не нравится клиентам в радиусе 1 км от моей точки» |

## 8. План работ — что сейчас в фокусе (9 фич MVP)

**Контекст:** обрыв пайплайна на pain_tags. План — достроить путь до сгенерированного письма + дать onboarding.

### Уровень 1 (без этого MVP не работает)
1. **Контакты компании в карточке** — связать `maps.companies` с краулером из `searches/`, для каждой компании дёргать её сайт → email/телефон → в карточку.
2. **Цитата-доказательство под каждой болью** — рядом с pain_tag показывать 1-2 ярких отзыва (через embedding similarity reviews → pain_tag centroid). Без этого «диагноз» не выглядит диагнозом.
3. **Драфт письма по кнопке** — компания + 2-3 цитаты её клиентов + email → LLM (новый ассистент `reviews_ai_outreach_draft`) → готовый драфт. **Aha-moment.**

### Уровень 2 (чтобы юзер не терялся)
4. **Лист лидов** — таблица `lead_lists` + `lead_list_items` (FK на companies). Кнопка «в список» на карточке, страница «Мои списки».
5. **Пресеты ниш на /app/leads** — кликабельные карточки «Стоматологии Москвы», «Автосервисы СПб» и т.п., клик — запуск готового поиска.
6. **Demo-результат на главной (read-only)** — публичная страница с заранее прогнанным поиском для незарегистрированных.

### Уровень 3 (связка с тем что уже есть)
7. **Maps → Outreach кампании** — из листа лидов кнопка «создать кампанию» → шаблон с переменными `{name}`, `{city}`, `{top_pain}`, `{pain_quote}` → отправка через существующий EmailService.
8. **SSE прогресс парсинга в UI** (доделать/проверить) — «парсю 23/100 компаний… анализирую отзывы…».

### Что НЕ берём сейчас (намеренно)
- Биллинг, юр. оферта (отдельно).
- Tenders, WhatsApp, voice-агенты, rank tracker, reputation monitoring.
- Глубокие дашборды.
- VK / Telegram OAuth.

## 9. Сценарий «первого успеха» (Aha-moment, целевой UX)

```
1. Юзер на /app/leads видит карточки-пресеты ниш
2. Клик «Стоматологии Москвы» → запускается поиск
3. SSE-стрим: «парсю 2GIS… 12/50… анализирую отзывы… выделяю боли»
4. Появляются карточки компаний:
   - название, рейтинг, кол-во отзывов
   - топ-боли: «Долгое ожидание» (5 отзывов), «Грязно» (3 отзыва)
   - под болью — цитата клиента
   - контакты: телефон, сайт, email (если найден)
   - кнопки: [В список] [Письмо]
5. Клик [Письмо] → LLM генерит драфт с упоминанием боли + цитаты
6. Клик [В список] → выбор/создание списка лидов
7. На странице «Мои списки» → кнопка [Создать кампанию] → outreach
```
