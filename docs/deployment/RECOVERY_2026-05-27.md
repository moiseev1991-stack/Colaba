> **Кому это:** человеку с доступом root@88.210.53.183 (подрядчик / админ).
> **Сколько займёт:** 5–15 минут.
> **Что произошло:** в ходе деплоя новой UI-фичи попробовали пересобрать
> frontend-образ прямо на VPS. У машины 3.8GB RAM, `npm ci + next build`
> съели всю память, OOM-killer прибил **Traefik и Coolify заодно со сборкой**.
> Сама ОС жива (ping проходит, SSH работает на 22), но HTTP-стек целиком лежит.

## Шаг 1 — поднять прод (быстро, без новой UI-фичи)

Зайди по SSH:

```bash
ssh root@88.210.53.183
```

И прогони готовый скрипт-реаниматор (он лежит в репо):

```bash
cd /opt/colaba-src && git fetch && git reset --hard origin/main
bash scripts/deployment/recover-after-oom.sh
```

Что он делает (если интересно):
1. Прибивает зависшие `npm`/`node`/`docker build` процессы (если ещё держат RAM).
2. Удаляет уже умершие контейнеры (статус `Exited`).
3. Поднимает все сервисы из `/opt/colaba/docker-compose.prod.yml` на **тех образах, что уже скачаны** на VPS (`--no-build --pull never`). Никакой сборки, никакого OOM.
4. Делает smoke-тест: проверяет что `https://spinlid.ru/` и `/api/v1/maps/cities` отвечают 200.

После этого прод вернётся к **тому состоянию, в котором был до моей попытки пересобрать фронт** (это коммит `0f14c4a` или близкий, backend на 1.6.0). Новая UI-фича маpс при этом **не появится** — для неё нужен новый frontend-образ (см. шаг 2).

Если skript не отработает (например compose ругнётся на отсутствие образов в локальном кэше) — fallback:

```bash
docker ps -aq --filter status=exited | xargs -r docker rm
pkill -9 -f "npm|node"
cd /opt/colaba && docker compose -f docker-compose.prod.yml up -d
```

Или ядерный вариант — просто **перезагрузить VPS** через панель хостера. У всех сервисов в `docker-compose.prod.yml` стоит `restart: unless-stopped` → они поднимутся автоматически на старых образах.

## Шаг 2 — выкатить новую UI-фичу маpс (после того как Шаг 1 поднял прод)

В main уже лежит коммит `b474d0c feat(maps-ui)` — переезд формы «По картам» на стиль «По сайтам», конструктор условий фильтра отзывов по тексту, переделанный drawer с поиском по тексту и подсветкой. **Бэкенд для неё уже легко собирается** на VPS (только Python, без npm) — RAM хватает с запасом.

### Шаг 2.1 — пересобрать только backend (это безопасно)

```bash
cd /opt/colaba-src && git fetch && git reset --hard origin/main
docker build -t ghcr.io/moiseev1991-stack/colaba-backend:latest /opt/colaba-src/backend
```

```bash
# Синхронизировать docker-compose.prod.yml в /opt/colaba (он не синкается автоматически)
cp /opt/colaba-src/docker-compose.prod.yml /opt/colaba/docker-compose.prod.yml
```

```bash
cd /opt/colaba && BACKEND_IMAGE=ghcr.io/moiseev1991-stack/colaba-backend FRONTEND_IMAGE=ghcr.io/moiseev1991-stack/colaba-frontend IMAGE_TAG=latest \
  docker compose -f docker-compose.prod.yml run --rm backend alembic upgrade head
```

```bash
cd /opt/colaba && BACKEND_IMAGE=ghcr.io/moiseev1991-stack/colaba-backend FRONTEND_IMAGE=ghcr.io/moiseev1991-stack/colaba-frontend IMAGE_TAG=latest \
  docker compose -f docker-compose.prod.yml up -d --force-recreate --pull never --no-build backend celery-worker
```

Это даст работающие новые **backend-фильтры отзывов по тексту** (через API), но фронт-форма «По картам» ещё будет старая.

### Шаг 2.2 — выкатить новый frontend БЕЗ риска OOM (собираем локально)

**На VPS НЕ собирать фронт.** Это и положило прод.

Есть два варианта.

#### Вариант A — Дима собирает у себя локально (через Docker Desktop на Windows)

В PowerShell на машине Димы:
```powershell
cd e:\cod\Colaba
pwsh scripts/deployment/build-frontend-local-and-push.ps1
```

Скрипт сделает: `docker build` → `docker save -o file.tar` → `scp` на сервер → `docker load` → `docker compose up -d frontend`. Нужен только SSH-доступ к серверу. Готово за 5-10 минут.

#### Вариант B — собрать на VPS, но добавив swap

Если по какой-то причине нельзя собирать локально, можно дать VPS виртуальной памяти на время сборки:

```bash
# Создаём 4GB swap-файл (одноразово, постоянно работать со swap для прод-нагрузки не надо)
fallocate -l 4G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
free -m  # должно показать swap: 4096

# Теперь билд переживёт, но будет медленный (~15-20 мин)
docker build -t ghcr.io/moiseev1991-stack/colaba-frontend:latest /opt/colaba-src/frontend

# После успешного билда — выключить swap, чтобы прод-нагрузка не уходила на диск
swapoff /swapfile
rm /swapfile
```

После любого варианта — пересоздать frontend контейнер:
```bash
cd /opt/colaba && BACKEND_IMAGE=ghcr.io/moiseev1991-stack/colaba-backend FRONTEND_IMAGE=ghcr.io/moiseev1991-stack/colaba-frontend IMAGE_TAG=latest \
  docker compose -f docker-compose.prod.yml up -d --force-recreate --pull never --no-build frontend
```

Открыть `https://spinlid.ru/app/leads` → таб «По картам» → должна быть новая форма с большой кнопкой «Найти компании» и расширенными настройками.

## Шаг 3 — дать Диме доступ, чтобы это не повторялось

Сейчас Дима не может починить даже простое падение, потому что у него нет ни SSH-доступа, ни доступа в панель хостера. Это блокер: если на сервере что-то упадёт ночью / в выходные / когда тебя нет — прод будет стоять.

Минимум что нужно дать:
1. **SSH-ключ или root-пароль** к 88.210.53.183.
2. **Логин/пароль от панели хостера** (нужен для перезагрузки, замены сервера, оплаты).

Идеально:
3. **Запись в его password manager-е** (1Password / Bitwarden) с обоими.

## Что НЕ нужно делать (на что не наступать)

- ❌ Не пересобирать фронт прямо на этом VPS без swap — OOM-killer прибьёт всё, включая Coolify-панель. Именно так мы и попали в этот инцидент.
- ❌ Не удалять `/opt/colaba/.env` — там ключи 2GIS, ProxyAPI, postgres-пароль.
- ❌ Не делать `docker volume rm postgres_data` — там вся БД (компании, отзывы, AI-теги, реальные пользовательские данные).
